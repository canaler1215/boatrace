### P4-β 撤退の本質的原因

「ROI が確定的に悪い」ではなく、**「skip_rate 99-100% で bet サンプルが貯まらない」**。
Day 5+6+7+8 = 540 races で 2 bets。`reeval_skip_simulation.py` で機械的 EV≥1.0
採用シミュレーションを試したが ROI -30.3%、MECH+BORDER 拡張も -17.3% で改善せず。
Claude の skip 判断は概ね合理的、機械化で改善せず、構造的に bet 機会増えない 3 点で撤退。

### 真の ROI を判定するためのサンプルサイズ

P4-α のサンプルサイズ分析: 真の ROI = +10% を 0% から区別するには **9,979 bets ≈ 200 race-days 必要**。
P4-β は 540 races / 2 bets で構造的にここに到達不能。

## 議論するポイント（Q-B）

前セッションでは選択肢として以下 3 つが提示された:

### (P-i) 完全凍結
P1-P3.5 のスクリプトはツールとして残すが、運用目標を捨てる。サンクコスト切り。

**根拠**: B 系列 4 系統（trifecta / win / place / condition）すべてで控除率を破る歪みを
発見できなかった以上、LLM が同じ市場で +EV を見つける構造的根拠が乏しい。

### (P-ii) 問題定義を変える
3 連単 ROI 最適化を捨て、別 KPI で再走。例:
- 単勝 / 複勝の上位推奨レース（ROI ではなく的中率優先）
- 特定条件レース（G1 / 特定場 / 特定気象）への限定運用
- bet ではなくレース選定のみ（人間が最終判断）

**コスト**: `/predict` 出力スキーマ変更 + 既存スクリプト改造。ガードレールゼロ原則の維持要検討。

### (P-iii) 観察モードで保留
スクリプトをそのまま週 1 程度で回し、bet サンプルを年単位で累積するだけ。来年再考。

**忘却リスク**: 1 年放置すると判断材料が冷えて、次の判断が「結局やめる」になりがち。

### (P-iv) 解説・原因分析専用モード（前セッションで議論済み）

価値は限定的。既存ドキュメントで重要な分析は済んでおり、凍結フェーズへの解析を増やしても
フィードバック先がない。LLM の narrative は検証なしに使うと判断を歪めるリスク。

**唯一ユニークに有用な使い方**: 「新軸を立てる前の事前 sanity check として、Claude に
RESULTS.md 群をまとめて読ませて『**まだ試していない仮説の列挙**』を依頼する」。これは
1 セッション 30 分 × 1 回の使い切り用途で、定常運用ではない。

### (P-v) ハイブリッド = (P-i) + (P-iii) （前セッション提案）

P スクリプトはツールとして保持、`/predict` を月数回任意で回すのは可だが、ROI 改善目標は持たない。
工数低、忘却リスク中。

## ユーザー視点で「LLM 予想で試したかったこと」のうち未実施の論点（前セッションで提示済み）

- **特定条件レース（G1 / 雨天 / 風速 5m+ 等）での Claude の判断力検証**
  - 前セッション最終ランで P-v `wind_speed_band` × `weather` の市場効率事前検証済み:
    **採用基準達成 0 / 16 valid segments**（最善 ev=0.95, CI 上限 1.02）。
  - **LLM を実走させる前の市場側下限が「LLM が edge を作れる余地ほぼなし」を示している**
  - **G1 / SG レース限定**は K ファイルから race grade 抽出が未実装（B ファイル要参照）。
    ただし年 300〜500 races / Claude bet 30〜50 件でサンプル不足致命的
  - **wave_height** は K ファイルに `波 1cm` 形式で存在するが現行 parser 未抽出
  - 「特定条件で LLM が edge を作れる」仮説の検証は **市場側下限を超える根拠が要る**
- **解説・原因分析用途**（上記 P-iv）
- **数千レース規模での Claude vs LightGBM 比較** — 既存モデルとの比較研究
